# domain 層（最内層 / エンタープライズビジネスルール）

このアプリで最も重要かつ安定した層。**技術的な詳細を一切含めない**。

- React も Next.js も DB も HTTP も知らない
- ここにあるのは「業務上の概念」そのもの
- 他のどの層にも依存しない（何も import しない）

## 置くもの

- **entities/** … 一意な識別子を持つ「モノ」。ライフサイクルを通じて同一性を保つ。
  例: `Prize`（賞）, `Wheel`（ルーレット盤）
- **value-objects/** … 値そのもので等価判定される不変オブジェクト。
  例: `Probability`（0〜1の確率）, `Color`
- **repositories/** … データ永続化の「インターフェース（契約）」だけを定義する。
  実装は infrastructure 層が担う（依存性逆転の原則 = DIP）。

## なぜインターフェースだけを置くのか

domain は「保存できること」を要求するが、「どう保存するか（localStorage? API?）」は
気にしない。契約だけを内側に置き、実装を外側に追い出すことで、
内側が外側に依存しない構造を保てる。
